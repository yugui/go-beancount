# Go Code Review Comments — Key Rules

Source: https://go.dev/wiki/CodeReviewComments

## Gofmt

Run `gofmt` (or `goimports`) on all Go code. This project's hooks run `gofmt` automatically on write.

## Comment Sentences

Comments documenting declarations should be full sentences beginning with the name being described and ending in a period:

```go
// Request represents a request to run a command.
type Request struct { ... }

// Encode writes the JSON encoding of req to w.
func Encode(w io.Writer, req *Request) { ... }
```

## Contexts

- `context.Context` should be the **first parameter** of functions that use it: `func F(ctx context.Context, ...)`
- **Never** store context in a struct field; pass it as a parameter instead
- Do not create custom Context types or use interfaces other than `context.Context`
- When in doubt, pass the context even if you think you don't need it

## Copying

- Be careful copying structs from other packages — may alias internal state
- Do not copy a value of type `T` if its methods are associated with `*T`

## Crypto Rand

- **Never** use `math/rand` or `math/rand/v2` to generate keys (even throwaway ones)
- Use `crypto/rand.Reader`; for text use `crypto/rand.Text` or encode with `encoding/hex`/`encoding/base64`

## Declaring Empty Slices

Prefer `var t []string` (nil slice) over `t := []string{}` (non-nil empty slice).
Exception: when encoding JSON (nil → `null`, `[]string{}` → `[]`).

## Doc Comments

All top-level exported names must have doc comments. Non-trivial unexported types and functions should too.

## Don't Panic

Do not use `panic` for normal error handling. Use `error` and multiple return values.

## Error Strings

- Error strings must **not** be capitalized (unless starting with a proper noun/acronym)
- Error strings must **not** end with punctuation
- Correct: `fmt.Errorf("something bad")` — Wrong: `fmt.Errorf("Something bad.")`
- This ensures `log.Printf("reading %s: %v", f, err)` reads naturally

## Goroutine Lifetimes

- Make it clear when — or whether — goroutines exit
- Goroutines can leak by blocking on channel sends/receives; GC will not collect them
- If goroutine lifetimes aren't obvious, document when and why they exit

## Handle Errors

- **Do not** discard errors using `_` variables
- If a function returns an error, check it
- Handle the error, return it, or in truly exceptional situations, panic

## Imports

- Avoid renaming imports except to avoid name collisions
- Group imports with blank lines between groups:
  1. Standard library
  2. Everything else (or split: project / third-party)
- `goimports` manages this automatically

## Import Blank (`import _`)

Import for side effects only in `package main` or in tests. Not in library packages.

## Import Dot (`import .`)

Do not use `import .` except in test files that cannot be part of the tested package due to circular dependencies.

## In-Band Errors

Do not return special values (`-1`, `""`, `nil`) to signal errors. Use multiple return values:

```go
// Good
func Lookup(key string) (value string, ok bool)

// Bad — caller can ignore the error signal
func Lookup(key string) string
```

## Indent Error Flow

Keep normal code path at minimum indentation; handle errors first and return/continue early:

```go
// Good
if err != nil {
    return err
}
// normal code here

// Bad
if err != nil {
    // error handling
} else {
    // normal code
}
```

## Initialisms

Acronyms and initialisms must have **consistent case** throughout:
- `URL` not `Url`, `HTTP` not `Http`, `ID` not `Id`, `appID` not `appId`
- `ServeHTTP` not `ServeHttp`
- Code generated by protoc is exempt

## Interfaces

- Interfaces belong in the **consuming** package, not the implementing one
- Implementing package should return **concrete types** (pointer or struct)
- Do not define interfaces for mocking on the producer side
- Do not define interfaces before they are used

## Line Length

No rigid limit. Break lines when they become semantically unwieldy, not purely for length. Long lines often indicate long names — fix the names.

## Mixed Caps

Use `MixedCaps`/`mixedCaps` even when it breaks conventions from other languages. `maxLength` not `MAX_LENGTH`.

## Named Result Parameters

Use named results to clarify when returning multiple values of the same type or when the meaning is unclear. Do not use just to avoid declaring a local variable.

## Package Comments

Must appear adjacent to `package` clause with no blank line:

```go
// Package math provides basic constants and mathematical functions.
package math
```

## Package Names

- Omit the package name from exported identifiers: `File` not `ChubbyFile` in package `chubby`
- Avoid meaningless names: `util`, `common`, `misc`, `api`, `types`, `interfaces`

## Pass Values

Do not pass pointers just to save bytes. If a function only reads `*x`, the argument should not be a pointer. Avoid `*string`, `*io.Reader` parameters.

## Receiver Names

- Use short (1-2 letter) abbreviation of the type name: `c` or `cl` for `Client`
- **Never** use `me`, `this`, `self`
- Be **consistent**: same receiver name across all methods of a type

## Receiver Type

- Prefer pointer receiver when: method mutates, struct has sync fields, receiver is large
- Use value receiver for: small read-only structs, basic types
- **Do not mix** pointer and value receivers within a type

## Synchronous Functions

Prefer synchronous functions (return results directly) over asynchronous ones. Callers can always add concurrency by calling from a goroutine.

## Useful Test Failures

Tests must produce helpful failure messages. Include: function name, inputs, actual value, expected value.

```go
// Good
t.Errorf("Foo(%q) = %d; want %d", tt.in, got, tt.want)
```

Order: **actual before expected** ("got X, want Y" not "expected Y, got X").

## Variable Names

- Short names for small scopes (`c`, `i`); longer names for wider scopes
- Proportional to scope: the further from declaration, the more descriptive
- Receiver names: 1-2 letters; loop indices: single letter; globals: descriptive
